# 车夫 / NPC「传送」与 Hook 思路（对照菜单回城 `4680`）

本文基于 `jadx/sources/defpackage` 下 **`ao.java`、`q.java`、`bt.java`、`bz.java`、`t.java`** 的反编译逻辑，说明：**与 NPC 对话选「传送」**时客户端走哪条链路、发什么包，以及**能否用 Hook 复用这条链路**代替「菜单回城 + fake `8199`」一类做法。

源码里**没有出现「车夫」字面**；车夫、驿站等均表现为带标准功能菜单的 **NPC**。NPC 可选功能文案来自全局表 **`t.I[]`**，其中下标 **`3`** 固定为 **`"传送"`**。

---

## 1. 与「操作菜单回城」的本质区别

| 维度 | 菜单「回城」 | NPC「传送」 |
|------|--------------|-------------|
| 客户端发包 | **`4680`（`0x1248`）**，`ao.aZ()` → `bz.C(4680, bt.ad)` | 确认目的地时发 **`4142`（`0x102E`）**，见 **`bz.a(..., 4142, ...)`**；对话入口常先发 **`4106`** |
| 典型消耗 | 回城券等（由服务端校验，失败常 **`8193`**） | 由服务端规则决定（银两/等级/是否开放该目的地等），与 **`4680` 不是同一套逻辑 |
| 切图表现 | 依赖服务端下发 **`8199/8245/...`**（见 `docs/回城逻辑分析.md`） | 同样依赖服务端后续下发地图类包；客户端**不会**因为发了 `4142` 就本地直接改图 |
| 数据准备 | 一般不要求客户端先有一份「目的地列表」 | 需要 **`bt.cs` / `bt.cr`**（**地图名称 + 目的地 id**），由收包 **`8227`** 填充 |

结论：**不能**认为「触发车夫传送」与「发 `4680`」在服务上等价；若要 Hook 车夫链路，目标是 **合法发出 `4142` 并让服务端按 NPC 传送规则处理**，而不是冒充 `4680`。

---

## 2. 整体报文流：`4106` 触发对话 → `8207` / `8227` → `4142` 传送

与抓包（如 **`log/回城分析日志2.out`**）及源码一致，可概括为：

1. **`4106`（`0x100A`）— 客户端请求与 NPC 对话**  
   - 构造：**`bz.c((short) 4106, bt.ad, (int) bt.t[this.af].a)`**（`bz.java` 约 506～523 行）。  
   - 内层帧顺序：**`writeShort(4106)`** → **`writeInt(npcId)`** → **`writeUTF(bt.b)`** → **`writeUTF(bt.d)`** → **`writeUTF(bt.ad)`**。  
   - 其中 **`npcId`** 即 **`bt.t[this.af].a`**：当前选中 NPC 在 **`az`** 上的**实体 id**（`az.a(DataInputStream)` 里第一个 **`readShort()`** 读入的字段 **`a`**）。  
   - 典型触发：玩家在地图上对 **NPC 且 `bt.t[this.af].e == 1`** 确认交互时（**`ao.java`** 约 2884～2890 行）。  
   - **另有一处** **`bz.c((short) 4106, bt.ad, -1)`**（**`ao.java`** 约 17360 行）：**无具体 NPC 实体**的对话入口，与「点车夫」场景不同。

2. **`8207`（`0x200F`）— 服务端返回 NPC 台词**  
   - 对话/剧情 **`readUTF`**，客户端用于展示。

3. **`8227`（`0x2023`）— 服务端返回可传送目的地列表**  
   - 由 **`bt.j`** 解析：每项 **`readShort` + `readUTF`** → **`cr[i]`**（**目的地 id**，本仓库抓包样本中与落地 **`8199` 的 `bt.ar`（地图 id）一致**，见 **§9.5**）与 **`cs[i]`**（**地图/目的地展示名**）。

4. **`4142`（`0x102E`）— 客户端确认传送**  
   - **`bz.a((short) 4142, bt.ad, npcIdOrMinus1, destinationMapId)`**  
   - **`writeInt`**：自 NPC 打开传送 UI 且 **`bJ==true`** 时为 **`bt.t[this.af].a`**，即**与 `4106` 相同的 npcId**；否则 **`-1`**。  
   - **`writeShort`**：**`bt.cr[选中行]`**，即 **`8227` 下发的目的地 id**（样本中即**目标地图 id**）。  
   - 之后服务端照常下发 **`8245/8199/…`** 完成切图（同 **`docs/回城逻辑分析.md` §9**）。

## 3. NPC 对话里「传送」选项如何进入

玩家在地图上对 **NPC 实体**打开对话后，可选菜单项来自 **`bH`（Vector）** 等 UI 状态；用户确认某一项时，在 **`ao.v(int i2)`**（`l == 0` 分支）里会把选中项文字与 **`t.I[]`** 逐项比对，得到下标 **`i3`**，再 `switch`。

与「传送」对应的是 **`case 3`**（即文案等于 **`t.I[3] == "传送"`**）：

- 若 **`bt.cr != null && bt.cr.length > 0`**：调用 **`a(true)`**，打开「传送目的地」列表界面。
- 否则：提示 **`"传送列表为空!"`**。

也就是说：**没有服务端事先下发的传送列表（`bt.cr`/`bt.cs`），点「传送」也不会进入选目的地流程。**

（`t.I` 定义见 `t.java`，`"传送"` 为数组第 4 个元素，下标 3。）

## 4. 传送列表 `bt.cs` / `bt.cr` 从哪来：收包 `8227`

`q.java` **`case 8227`**：

- 调用 **`bt.j(DataInputStream)`** 解析 payload。
- 若解析后 **`bt.cs == null`**：提示 **`"没有可传送的目的地!"`**。
- 部分界面状态下会调用 **`a.e.a(false)`**（壳层接口，用于刷新/打开与传送相关的 UI，与 `a(true)` 成对出现）。

**`bt.j` 协议（摘要）**：

1. **`readByte` → `n`**；若 `n <= 0`，调用 **`bt.l()`** 清空列表并返回。
2. 否则分配 **`cr = new short[n]`**，**`cs = new String[n]`**。
3. 对 `i = 0 .. n-1`：**`cr[i] = dataInputStream.readShort()`**，**`cs[i] = dataInputStream.readUTF()`**（注意顺序：**先 short 再 UTF**）。

含义（与玩家理解对齐）：

- **`cs[i]`**：**地图/目的地名称**（展示用 UTF 串）。
- **`cr[i]`**：**目的地 id**（`short`）。在本仓库**车夫传送抓包**中，与随后 **`8199` 的 `bt.ar` 一致**（**§9.5**）；其它玩法是否恒等于地图 id 需按区服验证。

### 4.1 `4142` 里填的是「地图 id」还是「8227 里的列表下标」？

源码结论（**不是下标**）：

- UI 当前高亮行由 **`this.e.ar.g()`** 得到，记为 **`g`**（0、1、2… 只是**客户端数组下标**）。
- 组 **`4142` 时写入的 `short` 是 **`bt.cr[g]`**，即 **8227 为第 `g` 行下发的那个 `readShort()` 的值**（样本中即**目的地/地图 id**）。
- **不会**把 **`g` 本身**当作 `writeShort`  payload 发出去（除非恰好 `cr[g]==g`，那只是数值巧合）。

因此「不同车夫同一序号对应不同目的地」在协议上的处理方式：

- A 车夫 `8227` 可能：`cr[0]=50`，`cs[0]="长安"`；B 车夫可能：`cr[0]=120`，`cs[0]="傲来"`。
- 用户都选第 0 项时，**`4142` 分别发 `50` 和 `120`**，**不是**都发 `0`。

Hook 时若要伪造 `4142`：应使用 **某次真实 `8227` 里对应目的地的那一行的 `cr[i]`**（或抓包得到的 `writeShort` 值），**不要**误以为要发列表下标。

---

## 5. 打开「传送目的地」界面：`ao.a(boolean z2)`

**`ao.a(boolean z2)`**（公开实例方法）在 **`z2 == true`**（NPC 传送入口使用）时大致行为：

- **`this.bJ = z2`**（为 **`true`** 时，后续 **`4142` 的 `writeInt` 会带 **npcId**（`bt.t[this.af].a`），与 **`4106`** 一致；见 **§6**）。
- 重置部分 UI 状态，`this.l = 0` 等。
- 标题 **`"传送目的地"`**，用 **`bt.cs`** 作为列表文案初始化列表控件 **`this.e.ar`**。
- 将 **`this.k` / `this.j` 置为 `10`**，进入与列表、确认键相关的输入处理分支（与 **`w(int)`** 等配合）。

**`a(false)`** 在 **`8227`** 分支里也会出现，用于另一类界面刷新；此时 **`bJ` 为 false**，**`4142` 的 `writeInt` 可为 `-1`**（与带 npcId 的路径区分）。

---

## 6. 用户确认目的地：发包 `4142`

在 **`ao.w(int i2)`** 中，当焦点在传送列表且用户按下确认类键（`268435456` / `1073741824` / `517`）时：

- 调用 **`bz.a((short) 4142, bt.ad, (int)(this.bJ ? bt.t[this.af].a : (short) -1), bt.cr[this.e.ar.g()])`**。
- 将返回的 **`byte[]`** 封装为 **`w((short) 4142, bArr)`**，**`a.i.a(...)`** 发往网络。

**`bz.a(short, String, int, short)`**（实现约 `bz.java` 701 行起）内层帧字段顺序为：

1. 帧头 **`0xAE`（-82）**
2. **`writeShort(4142)`**
3. **`writeInt(i)`**：**npcId** = **`bt.t[this.af].a`**（与 **`4106`** 中 **`writeInt`** 相同），若 **`bJ` 为 false** 则为 **`-1`**
4. **`writeShort(s2)`**：**目的地地图 id**（协议上为 **`8227` 下发的 `cr[选中行]`**；本仓库车夫样本中等于 **`8199` 的 `bt.ar`**）
5. **`writeUTF(bt.b)`、`writeUTF(bt.d)`、`writeUTF(bt.ad)`**（会话与角色 id）
6. 帧尾 **`0xAF`（-81）**

因此，一次「等价于点选传送并确认」的客户端行为，至少要满足：

- **`bt.ad`** 等会话字段有效；
- **`this.e.ar.g()`** 为合法行号，且 **`bt.cr`/`bt.cs` 已按相同行对齐**；
- **`this.bJ`、`this.af`、`bt.t`** 与打开传送界面的路径一致（否则 **`writeInt`（npcId）** 可能与真实对话上下文不一致，服务端易拒包）。

## 7. 能否 Hook「车夫传送」代替 fake `8199`？

### 7.1 思路 A：只伪造发送 `4142`（最小发包）

在合适的时机直接构造 **`bz.a(4142, bt.ad, npcIdOrMinus1, destinationShort)`** 并入队发送。

**优点**：不碰地图包伪造；若服务端校验宽松，可能直接走正常切图下发。

**风险（需服务端验证）**：

- 服务端很可能校验：**是否刚与允许传送的 NPC 交互**、**是否在合法地图**、**该 `destinationShort` 是否属于当前 NPC 下发的列表**、**是否已扣费**等。
- 若校验失败，可能 **静默失败、弹 `8193`、或踢线**；与 fake `8199` 的「纯客户端改状态」风险剖面不同。

### 7.2 思路 B：复用客户端完整调用链（`a(true)` → 设置高亮 → `w(确认键)`）

尝试在运行时调用 **`ao.a(true)`**，再设法设置 **`this.e.ar`** 高亮行并调用 **`w(确认键值)`**。

**难点**：

- **`ao` 状态机**依赖 **`this.l`、`this.k`、`this.j`、`this.e.aq`** 等大量字段，少一步可能进错分支或空指针。
- **`this.af`**（NPC 下标）必须与 **`bt.t`**、当前对话上下文一致，否则 **`4142` 内 npcId** 可能错误。

适合作为**实验**，不适合作为稳定方案，除非严格对照当前帧 UI 状态复现。

### 7.3 思路 C：在真实与车夫对话后 Hook「确认」

玩家在正常运行中已与车夫交互、`8227` 已把 **`bt.cr/bt.cs`** 拉到内存，此时 Hook **仅替换目标行号**再让原有 **`w()`** 发送 `4142`。

**优点**：npcId、会话、列表来源与真实流程一致，服务端通过率相对最高。

**缺点**：仍需先处于**有车夫、有列表**的场景；不能凭空在副本地图「一键回主城」除非该图本就有传送 NPC 且列表含主城。

## 8. 易混淆：附近玩家菜单里的 `4110` / `4111`

在 **`ao` 其它 `l` 分支**（如附近玩家、队伍相关）中会出现 **`bz.a(4110, ...)`**、**`bz.b(4111, ...)`**，参数里带 **`bt.gH[...]`** 或 **`bt.q[...].b`** 等，语义偏向 **私聊/队伍操作**，**不是** NPC 传送确认包 **`4142`**。

分析「车夫传送」时应以 **`4106` → `8207`/`8227` → `4142`** 及 **`t.I[3]` → `a(true)`** 为准，不要与 **`4110/4111`** 混用。

## 9. 附录：`log/回城分析日志2.out` 车夫传送抓包实录

以下根据仓库内抓包文件 **`log/回城分析日志2.out`**（车夫传送流程）整理，用于与 **§2、§4、§6** 对照。**本附录仅描述该次会话中观察到的字节与顺序**，不同区服/版本以实测为准。

### 9.1 总流程（两次传送）

两次传送均呈现相同骨架：

1. 客户端发 **`4106`** — payload 内 **`writeInt`**（npcId）与随后 **`4142` 内第一个 `int` 完全一致（见 **§9.4**）。
2. 服务端回 **`8207`** — NPC 台词 UTF。
3. 服务端回 **`8227`** — **`bt.j`** 填充 **`bt.cr[]` / `bt.cs[]`**（**地图名 + id**）。
4. 客户端发 **`4142`** — **npcId + 目的地地图 id**（`cr[选中行]`）。
5. 服务端 **`8192` 容器**内跟随 **`8264`、`8265`**，再 **`8245` → 双 `8199` → `8198` → `8200`**（与 `docs/回城逻辑分析.md` §9 所述切图包序列一致）。

### 9.2 第一次：`8227` 仅 1 条目的地 → 落地 **豆仙庭（`mapId=78`）**

**`8227` payload（14 字节）** 与 **`bt.j` 解析**一致：

| 偏移 | 十六进制 | 含义 |
|------|-----------|------|
| 0 | `01` | `n=1` 条 |
| 1–2 | `00 4E` | `cr[0]=78` |
| 3–4 | `00 09` | UTF 长度 9 |
| 5–13 | `E8 B1 86 …` | **`cs[0]="豆仙庭"`**（UTF-8） |

**`4142` 内层（`47` 字节总长约）** 在 `AE … AF` 之间：

- **`writeShort(4142)`** → `10 2E`
- **`writeInt`** → `00 00 01 91`（**401** = npcId，与同次 **`4106`** 中 int 一致）
- **`writeShort`** → `00 4E`（**78** = 目的地地图 id）
- 随后三个 **`writeUTF`**：`bt.b`、`bt.d`、`bt.ad`

**落地 `8199`（第二包，目标图）**：`00 4E` + 9 字节 UTF **豆仙庭** + 坐标等 → **`bt.ar=78`** 与 **`8227` 的 `cr[0]`** 一致。

### 9.3 第二次：`8227` 共 14 条目的地 → 选 **`cr=14`** → 落地 **傲来广场**

**`8227` payloadLen=204**：首字节 **`0x0E`=14** 条。前几项示例：

| 行 | `cr`（short） | `cs`（展示名，摘要） |
|----|---------------|----------------------|
| 0 | `78` | 豆仙庭 |
| 1 | `4` | 长寿村 |
| 2 | `14` | 傲来广场（UTF 长度 `0x0C`=12） |
| 3 | `20` | 龙宫广场 |
| … | … | …（共 14 行） |

**`4142`**：`writeInt`=`00 00 1E 79`（**7801** = npcId），`writeShort`=`00 0E`（**14** = 目的地地图 id）。**7801** 与同次 **`4106`** 中的 int **一致**。

**落地 `8199`（第二包）**：`00 0E` + UTF **傲来广场** + **`00 F0 01 30`** → 坐标 **(240, 304)**。

**结论（本段样本）**：与 **§4.1** 一致——**发的是 `cr` 数值，不是列表下标**。

### 9.4 `4142` 第一个 `int` 与 `4106` 的关系（强相关）

| 次序 | `4106` 中 `int`（大端） | 随后 `4142` 中 `int` |
|------|-------------------------|----------------------|
| 第一次 | `0x191`（401） | `0x191`（401） |
| 第二次 | `0x1E79`（7801） | `0x1E79`（7801） |

即 **npcId**：与 **`bt.t[this.af].a`**、**`ao.w` / `bz.a(4142,…)`** 中 **`writeInt`** 一致，**不是地图 id**。

### 9.5 `cr` 与 `8199` 的 `bt.ar`（本日志范围内的结论）

在本文件**两次成功传送**中，**`4142` 的 `writeShort` 均等于落地后目标 `8199` 的 `bt.ar`**，且与 **`8227` 中对应行的 `readShort()`** 相同。其它区服/玩法是否恒成立需单独验证。

## 10. 小结与建议

1. **车夫链路**：**`4106(npcId)`** → **`8207` 台词** + **`8227`（地图名 `cs` + 目的地 id `cr`）** → 选「传送」打开列表 → **`4142(npcId, 目的地地图 id)`** → 服务端下发 **`8245/8199/…`**。
2. **菜单回城 `4680`** 与 **`4142` 传送**是两套请求；想用「车夫逻辑」绕开回城券，本质是 **让服务端走传送校验与扣费规则**，而不是复用 `4680` 响应。
3. **Hook 可行性**：技术上可以 **直接发 `4142`** 或 **嵌套调用 `ao` 方法**；是否达到与回城相同的效果 **完全取决于服务端是否承认这次 `4142`**。较现实的路线是 **思路 C（在真实列表已存在时改选目的地或自动确认）** 或 **思路 A + 抓包验证**。
4. 若服务端要求严格，**车夫 Hook 无法替代**「完整真实切图收包序列」或 **合法 `4680` + 真服下发**；此时仍应参考 **`docs/回城逻辑分析.md` §9** 的地图包与 **`4680` 文档**综合处理。
5. **车夫传送落地包序列**可与 **`log/回城分析日志2.out`** 对照（**§9**）：与菜单回城 fake 所缺的 **`8245/8199/…`** 属于**同一类服务端切图下发**，差异仅在**触发命令是 `4142` 还是 `4680` 等**。
