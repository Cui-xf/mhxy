# 网络包协议分析（基于 `log.out` 与反编译源码）

本文档结合 Hook Agent 抓取的 `log.out` 与 `jadx/sources/defpackage` 中 `av`、`at`、`q`、`bt`、`bz` 等类的逻辑，说明**业务包 ID**的含义、与日志的对应关系，以及 **payload 的解析顺序**。

---

## 1. Hook 日志里看到的是什么

- Agent（`NetSendHook` / 收包侧同类逻辑）打印的是类 `**w`** 解包后的视图：
  - `**id`**：`w.a()` 返回的 `short`，即业务包类型（十进制与十六进制并列出现在日志中）。
  - `**payload**`：字段 `**w.b**`（`byte[]`），即**已去掉 Socket 外层帧**后的内容。
  - `**childCount`**：字段 `**w.c`**（`Vector`）长度；仅当 `**id == 8192 (0x2000)**` 时表示「批量子包容器」。
- **发送包**的 `payload` 往往以 `**0xAE`** 开头、`**0xAF`** 结尾：这是 `bz` 里构造的内层「明文业务帧」，再由 `at` 加上长度等发往 Socket（详见下文第 2 节）。
- **接收包**的 `payload` **不包含** `AE/AF`，直接按 Java `**DataInputStream`** 的读写顺序解析（`readUTF` 为 Java 修改版 UTF-8，带 2 字节长度前缀）。

权威收包入口：`av.a(InputStream)`（`jadx/sources/defpackage/av.java`）。  
权威分发入口：`q.a(w)` 的 `switch (this.d)`（`jadx/sources/defpackage/q.java`）。

## 2. Socket 传输帧（与日志的层次关系）

日志中的 `**id` / `payload` 对应 `w`**，位于下列格式**之内层**：

### 2.1 明文帧（首字节 `0x02`）

```
0x02 | 包ID (short, BE) | 数据长度 (int, BE) | 数据 [N]
```

当 **包 ID = 8192** 时，数据区为批量子包（见 2.3）。

### 2.2 压缩帧（首字节 ≠ `0x02`）

```
压缩长度 (2 字节 BE) | zlib 压缩体
→ ci.a() 解压 → 与 2.1 相同的 [包ID | 长度 | 数据]
```

### 2.3 批量子包（ID = 8192）

```
子包数量 (1 字节)
重复 N 次: 子包ID (short BE) | 子包长度 (int BE) | 子包数据
```

客户端对子包逐个 `q.a(子 w)` 处理（`q.java` case 8192）。

### 2.4 发送侧拼接（`at.a(Vector)`）

首连接可带握手字节 `**bz.a` / `bz.b**`（如 `0x5A 0x48`），随后每个 `**w**`：

```
[w.b[0]] | [长度高字节] [长度低字节] | w.b[1..]
```

即 `**w.b` 第 0 字节原样发送**，再发 `**w.b.length - 1`** 作为 16 位长度，其余拷贝跟进。

## 4. 典型 payload 解析步骤（与日志逐字节对照）

### 4.1 发送 0x1900 登录（`log.out` 首包）

`w.b` 整段为 `bz` 写出：

1. `0xAE`
2. `writeShort(6400)`
3. `writeByte` 模块类型
4. `writeInt` ×2（`bb.a` / `bb.b`）
5. `writeUTF` ×2 设备信息
6. `writeInt(-1)`，`writeByte` ×3，`writeInt` 内存
7. `writeByte` 操作码（1 登录 / 2 注册）
8. `writeUTF`：`"登录类型:账号:密码"` 明文
9. `0xAF`

### 4.2 发送 0x109A (4250) 选角

`bz.c((short)4250, str)`（`bz.java`）：

1. `0xAE`，`writeShort(4250)`
2. `writeUTF(bt.b)`，`writeUTF(bt.d)`（会话/账号侧字符串，日志中为 `cc123456` 与 token）
3. `writeUTF(str)`（角色 id，如 `id_12_63775455`）
4. `0xAF`

### 4.3 接收 0x2002 (8194)

```
readUTF → bt.b
readUTF → bt.d
```

日志中前两字段即账号相关字符串与 token。

### 4.4 接收 0x2007 (8199) 地图/坐标摘要 — `bt.c` 与 `q.b()`

**包含义**：`q.java` **case 8199** 调用私有方法 **`b()`**：先 **`bt.c(DataInputStream)`** 把地图 id、名称、角色像素坐标等写入静态字段 **`bt.ar` / `bt.ae` / `bt.at` / `bt.au` / `bt.av` / `bt.aw` / `bt.bw` / `bt.bx`**，再根据是否换图、队伍状态等更新 **`a.e.I`**（地图渲染对象）、触发 **`a.e.e()` / `f()`**、**`a.e.I.d()`** 等。**因此 8199 不只是「一行数据」，而是客户端切场景/同步自己在地图上位置的入口之一**；完整贴图往往还依赖后续 **8200、8260** 等资源包（见 4.6、4.7）。

**Payload 与 `DataInputStream` 读序**（与 `bt.java` `c(DataInputStream)` 一致，全为大端）：

| 顺序 | 类型 | 写入 `bt` 字段 | 说明 |
|------|------|------------------|------|
| 1 | `short` | `bt.ar` | 地图/场景 id |
| 2 | `UTF` | `bt.ae` | 地图名（Java `readUTF` / `writeUTF`，2 字节无符号长度 + Modified UTF-8；常用汉字与 UTF-8 一致） |
| 3 | `short` | `bt.at` | 角色 X（像素或格坐标，与服务端约定一致） |
| 4 | `short` | `bt.au` | 角色 Y |
| 5 | `byte` | `bt.av` | 朝向/层等标志（随玩法使用） |
| 6 | `byte` | `bt.aw` | 同上 |
| 7 | `byte` | `bt.bw` | 同上 |
| 8 | `byte` | `bt.bx` | 同上 |

**Payload 长度**：`2 + (2 + UTF8字节数) + 2 + 2 + 4 = 12 + UTF8字节数`（`UTF` 段 = 2 字节长度前缀 + 正文）。

**实录解析示例 1**（`payloadLen=24`）：

```text
00 01 00 0C E7 A9 BA E9 97 B4 E9 80 9A E9 81 93 FF F6 FF F6 00 00 00 00
```

- `00 01` → `bt.ar = 1`
- `00 0C` + 12 字节 → `bt.ae = "空间通道"`
- `FF F6` → `bt.at = -10`（有符号 short）
- `FF F6` → `bt.au = -10`
- `00 00 00 00` → `bt.av`～`bt.bx` 均为 `0`

**实录解析示例 2**（`payloadLen=21`）：

```text
00 4E 00 09 E8 B1 86 E4 BB 99 E5 BA AD 01 00 01 E0 01 00 00 00
```

- `00 4E` → `bt.ar = 78`
- `00 09` + 9 字节 → `bt.ae = "豆仙庭"`
- `01 00` → `bt.at = 256`
- `01 E0` → `bt.au = 480`
- `01 00 00 00` → `bt.av=1`，`bt.aw`～`bt.bx=0`

构造 **`fakePayload`** 时应用 **`DataOutputStream` + `ByteArrayOutputStream`** 按上表 **`writeShort` / `writeUTF` / `writeByte`** 顺序写出，装入 **`w((short)8199, bytes)`** 再送入收包队列即可（详见 `docs/回城逻辑分析.md` §8）。

#### 4.4.1 为何常见「两个 8199 成对出现」？是否必须两条？

**客户端不要求成对。** `q.a(w)` 里 **case 8199** 对 **每一个** 包各执行一次 **`b()`**：每次都完整跑 **`bt.c` + 换图/坐标刷新**。代码里 **没有**「必须收到两条 8199 才生效」的状态机。

**成对现象来自服务端习惯用法**（从你提供的样本与 `log.out` 多段可归纳）：

| 次序 | 典型内容 | 作用（推断） |
|------|----------|----------------|
| **第 1 个 8199** | 固定多见：`ar=1`，`ae="空间通道"`，`at=au=-10`，后四字节全 `0`（24 字节 payload） | **过渡/加载用场景**：把角色先挂到专用地图 id 1，坐标占位（-10），便于卸载旧图资源、再下发目标图 |
| **第 2 个 8199** | `ar`、`ae`、坐标随目的地变化（多为 21 字节级，因地图名为 3 个汉字） | **真实目的地**：如「豆仙庭」「花果山」「长寿村」「小木屋」等 |

**你补充的一组示例**（第二包）解析：

```text
00 07 00 09 E8 8A B1 E6 9E 9C E5 B1 B1 01 30 02 00 00 00 00 00
```

- `bt.ar = 7`，`bt.ae = "花果山"`，`bt.at = 0x0130 = 304`，`bt.au = 0x0200 = 512`，`bt.av`～`bt.bx = 0`。

与 **上一组「豆仙庭」** 对比：**第一个 8199 与「空间通道 / -10,-10」包完全一致**；**仅第二个 8199 随目标地图变化**（id、名、xy、以及有时 `bt.av` 非 0）。

**Hook / 伪造时建议**：

- **只发一条 8199**：发 **第二个**（真实目标图 + 合法坐标）通常即可驱动客户端落到目标场景；若遇贴图/状态异常，再按服务端习惯 **先发过渡包、再发目标包** 复现完整序列。
- **完整模仿服务端**：连续注入 **两条**（先 24 字节「空间通道」，再目标 21 字节），并视需要继续跟 **8200 / 8260** 等（`log.out` 里双 8199 后常出现 **8200**）。

### 4.5 接收 0x2005 (8197) 角色属性 — `bt.d`

大块连续字段：`readUTF/readShort/readByte/readLong/...` 填充角色名、等级、气血魔法、各类资源与装备摘要等（`bt.java` `d(DataInputStream)` 起始约 1176 行）。解析需严格按顺序与类型，缺一不可。

### 4.6 接收 0x2035 (8245) 资源分块 — `bt.aI`

```
readShort → s2   // 资源块类型/编号
readShort → s3   // 替换目标编号；-1 表示不删除旧块
若 Vector 中不存在 s2:
    readInt  → L
    read L 字节 → 原始数据（多为 gzip：魔数 1F 8B）
若 s3 >= 0：从 Vector 移除编号为 s3 的块
```

日志中 `1 | -1` / `14 | -1` 即 `s2` 与 `s3`；随后 `readInt` 为压缩体长度，`1F 8B` 为 gzip。

### 4.7 接收 0x2044 (8260) 地图数据 — `q` case 8260 + `bt.aJ`

**前半段（可选大图）：**

```
readInt i5
若 i5 > 0:
    new byte[i5]
    readUTF → bt.kX   // 名称/键，日志中可见 "map"
    readFully(bArr)   // 长度 i5 的原始字节（常与地图主资源相关）
    构造 bu(bt.kX, bArr)
```

**后半段 `bt.aJ`：**

```
readShort n
重复 n 次:
    readInt   → 块 id
    readInt   → 块长度
    read 块长度 字节 → 单块数据
```

日志中 `0x2044` 超大 payload 即「可选整图 + 多分块」拼接而成。

### 4.8 接收 0x2219 (8729) — `bt.aF`

按顺序读取：数量、多组 `readInt`、多段 `readUTF`/`readByte`（角色秀、染色、称谓等 UI 数据）。末尾常跟一段中文（如 NPC 提示）。

### 4.9 接收 0x221A (8730) — `bt.at`（合成列表）

```
readByte n
重复 n 次:
    readShort, readUTF, readByte,
    readShort×2, readLong×2, readUTF×2,
    readByte m
    若 m > 0: m 组 (readUTF, readShort, readShort)
```

### 4.10 发送 0x1005 (4101) 心跳

`bz.a(short, String, int, int, short)` 使用固定 `writeShort(4101)`：

```
AE 10 05 | writeShort x | writeShort y | writeShort 朝向等 |
writeUTF(bt.b) | writeUTF(bt.d) | writeUTF(str) | AF
```

## 6. 自行扩展 ID 表的方法

1. 在 `jadx/sources/defpackage/q.java` 中搜索 `**case 十进制ID:`**（如 `case 8245:`）。
2. 查看该分支调用的 `**bt.xxx(DataInputStream)`** 或内联 `read*`。
3. 到 `bt.java`（或 `az.java` 等）定位对应 `static void` 方法，按 `**DataInputStream` 调用顺序**列出字段。

若 `q.a(DataInputStream)` 等方法反编译失败，需用 JVM 字节码工具或运行时 Hook 补全。
