# 战斗攻击失败排查分析

## 问题描述

游戏启动后在地图上触碰怪物可以进入战斗界面，但点击攻击后直接退出战斗，服务器提示：

> 战斗错误，当前战斗不存在

---

## 关键变量说明

| 变量 | 类型 | 含义 |
|------|------|------|
| `GlobalStatus.v` | `long` | 当前战斗 ID，发攻击包时携带，服务器用于查找战斗 |
| `GlobalStatus.w` | `long` | 战斗 ID 暂存，进战斗时赋给 `v` |
| `bq_1.g` | `static int` | 初始值 `-1`，理论上应在需要从服务器更新 `v` 时设为正数 |

---

## 战斗进入的两条路径

### 路径 A：服务器推送（正常路径）

```
服务器推送 opcode=8240
  → q.java: GlobalStatus.z(this.b)
      → v = w = var0.readLong()   ← 从服务器数据读取真实 fightId
  → MainCanvas.e.f((byte)1)
      → ao_1.f(1)
          → new bq_1(...)
              → GlobalStatus.v = GlobalStatus.w   ← v 为正确的 fightId
```

**结果**：`GlobalStatus.v` = 真实 fightId，攻击包携带正确 ID，服务器可找到对应战斗。

### 路径 B：本地直接进入（问题路径）

```
玩家在地图触碰怪物（ao_1.d() 方法，约第 2675 行）
  条件：GlobalStatus.bs != 0 || GlobalStatus.s != 0（有组队/PK 状态）
        && this.I.a(this.aO) 返回怪物对象
        && var2.b != null && var2.b.length != 0

  → GlobalStatus.w = 0L             ← 直接置零，不等服务器
  → 构建 GlobalStatus.N（怪物数组）
  → 构建 GlobalStatus.I（战斗角色数组）
  → 调用 this.d()
      → 判断 GlobalStatus.I != null && GlobalStatus.N != null 为真
      → this.f((byte)0)
          → new bq_1(...)
              → GlobalStatus.v = GlobalStatus.w = 0L   ← v 为 0！
```

**结果**：`GlobalStatus.v = 0L`，没有向服务器发送任何战斗进入请求（无 opcode=4195 包），服务器不知道这场战斗的存在。

## 攻击包发送逻辑

**调用位置**：`bq_1.java` 约第 1955 行

```java
NetPayloadBuilder.a((short)4180, GlobalStatus.ad, GlobalStatus.v, g, t, ...)
```

**包结构**（`NetPayloadBuilder.java` 约第 1624 行）：

```
writeByte(-82)            // 0xAE 包头
writeShort(4180)          // opcode
writeLong(GlobalStatus.v) // ← 战斗 ID，服务器用此查找战斗
writeInt(g)               // 技能槽
writeByte(t)              // 技能参数

```

若 `GlobalStatus.v = 0`，服务器收到 fightId=0，找不到对应战斗，返回错误提示。

## `bq_1.g` 永远为 -1 的问题

`q.java` 约第 2051 行有一段备用的战斗 ID 更新逻辑（处理 opcode=8206 包）：

```java
private static void b(DataInputStream var0) throws IOException {
    long var1 = var0.readLong();
    if (bq_1.g > 0 && GlobalStatus.v <= 0L) {  // 条件永远不成立！
        GlobalStatus.v = var1;
        bq_1.g = -1;
    }
    ...
}
```

`bq_1.g` 的初始值为 `-1`（`bq_1.java` 第 28 行），**整个代码库中从未有地方将其设为正数**，导致此更新逻辑永远不会执行。

## opcode=8218 的退出逻辑

`q.java` case 8218（战斗结束/错误响应包）：

```java
long var6 = this.b.readLong();  // 服务器的 fightId
byte var44 = this.b.readByte();
if (var6 == -1L) {
    // 服务器通知战斗结束
    GlobalStatus.v = -1L;
    MainCanvas.e.g.l();  // 退出战斗
} else if (bq_1.g <= 0 && var6 != GlobalStatus.v) {
    // 服务器 fightId 与本地 v 不一致 → 退出
    GlobalStatus.v = -1L;
    MainCanvas.e.g.l();
} else if (var44 == 0) {
    GlobalStatus.v = -1L;
} else {
    GlobalStatus.v = -1L;
}
```

若本地 `v=0` 而服务器返回真实 fightId，则 `var6 != GlobalStatus.v` 为真，直接触发退出战斗。

## 根因总结

| # | 根因 | 位置 |
|---|------|------|
| 1 | 本地触怪进入战斗时 `GlobalStatus.w = 0L`，未向服务器请求战斗 ID | `ao_1.java` 约第 2683 行 |
| 2 | `bq_1` 构造函数直接 `v = w`，若 `w=0` 则 `v=0` | `bq_1.java` 第 58 行 |
| 3 | 攻击包 4180 携带 `fightId=0`，服务器找不到对应战斗 | `bq_1.java` 约第 1955 行 |
| 4 | `bq_1.g` 从未被置为正数，8206 包的备用 `v` 更新逻辑永远不执行 | `bq_1.java` 第 28 行，`q.java` 第 2051 行 |
