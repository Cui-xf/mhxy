# 回城逻辑分析（`ao.aZ()` / 包 `0x1248`）

本文说明客户端从打开「操作」菜单到发送回城请求、以及服务端失败/成功时客户端行为的代码路径，依据 `jadx/sources/defpackage/ao.java`、`bz.java`、`q.java`、`bt.java`。

---

## 1. 入口：不是只有 `aZ()` 才叫「回城逻辑」

真正发起 **回城网络请求** 的是 **无参** `private void aZ()`（`ao.java` 约 17369 行），它会发 **包 ID `4680`（`0x1248`）**。

类中另有一个 **`private void aZ(int i2)`**（约 19153 行），用于 **符文嵌入/挖出** 等界面，与回城 **无关**。讨论回城时只看 **`aZ()`**。

---

## 2. UI 如何走到 `aZ()`

### 2.1 打开操作菜单

在 `ao` 某段按键处理里（约 2836～2858 行），当按键值为 **`536870912`（`0x20000000`）** 且满足 `bt.H == null && this.e.b == 0` 等条件时，会构造菜单并弹出：

- 第一项固定为 **`"回城"`**（`vector.addElement("回城")`）。
- 后续项包括「自动刷怪」「属性配点」（仅当 **`bt.bf != 0`**）、队伍相关（仅当 **`bt.bs >= 0`**）、「查看邮箱」「活动推荐」「传送」等。

`bt.bf` 来自角色包 **`bt.d(DataInputStream)`** 中的 **`readShort()`**（`bt.java` 约 1212 行），与 **可分配属性点** 相关；为 0 时不显示「属性配点」，菜单项下标会整体前移。

弹出菜单后设置 **`this.k = this.j = 100`**，并 **`bt.gQ = 1`**。

### 2.2 选中「回城」并确认

菜单交互由 **`ca`**（全局 UI 状态）记录当前高亮项 **`ca.o`**。用户按确认键后进入 **`aE(int i2)`**（约 17220 行）。

在 **`this.l == 0`**（非子对话框）且 **`i2` 为确认类键值**（`268435456` / `1073741824` / `517`）时：

- 若 **`bt.bf == 0`**（无「属性配点」项时菜单映射较简单）：
  - **`bt.bs < 0`**（无队伍菜单项）：**`ca.o == 0`** → **`aZ()`**；**`ca.o == 1`** → **`ba()`**（发 **`4659`**）；**`ca.o == 2`** → **`d((byte)0)`**（邮箱）等。
  - **`bt.bs >= 0`**：**`ca.o == 0`** 仍为 **`aZ()`**；**`ca.o == 1`** → **`ba()`**；**`ca.o == 2`** 会走 **`bd()`/`bc()`**（队伍）等。

- 若 **`bt.bf != 0`**，同一 **`ca.o`** 数字对应 **不同的菜单文字**（因为中间插入了「属性配点」），分支在 17261 行起另一套 `if (ca.o == …)`，但 **「回城」始终对应第一个选项 `ca.o == 0`**，仍调用 **`aZ()`**。

因此：**在标准「操作」菜单里选第一项并确认，就会调用 `aZ()`**（在常见配置下该项就是「回城」）。

## 3. 发包：`0x1248`（4680）格式

`aZ()` 实现（约 17369～17377 行）：

```java
byte[] bArrC = bz.C((short) 4680, bt.ad);
a.i.a(new w((short) 4680, bArrC));
this.e.a((String) null);
```

`bz.C(short s, String str)`（`bz.java` 约 4701 行）写出内层帧：

| 顺序 | 内容 |
|------|------|
| 1 | `0xAE` |
| 2 | `writeShort(s)` → `4680` |
| 3 | `writeUTF(bt.b)` | 帐号类会话串 |
| 4 | `writeUTF(bt.d)` | 会话 Token |
| 5 | `writeUTF(str)` → **`bt.ad`** | 当前角色 id（如 `id_12_63775455`） |
| 6 | `0xAF` |

与日志完全一致：`AE 12 48` + 三个 UTF + `AF`。

## 4. 失败：你提供的日志（回城券不足）

### 4.1 收包：`0x2001`（8193）

服务端返回 **包 ID `8193`**，payload 由 **`bt.w(DataInputStream)`** 解析（`bt.java` 约 2542 行）：

1. **`readByte()` → `bt.eB`**
2. 若 **`eB <= 0`**（按有符号字节理解）：**`readUTF()` → `bt.eC`**（服务端自定义全文）
3. 若 **`eB > 0`**：`eC = t.Y[eB]`（走本地预置文案表）

你的 hex：`81 00 18 E5 9B 9E…`

- 第一字节 **`0x81`** 作为 **`readByte()`** 为 **`-127 ≤ 0`**，走 **`readUTF()`** 分支。
- 后续为 UTF 长度与内容，解码即 **`回城券数量不足！`**（日志里 `readUTF` 已打印）。

### 4.2 客户端如何提示

`q.java` **case 8193**（约 158～191 行）在排除若干特殊 `eB`（如 53、48、-2 等）后：

- 若 **`bt.eC != null`**：**`this.a.b(bt.eC)`** → 用服务端字符串 **弹提示/气泡**。
- 否则用 **`t.Y[eB]`**。

因此：**回城失败时不会单独写 `case 4680`，统一走「系统消息包」8193。**

---

## 5. 成功：券足够时客户端会怎样？

客户端 **没有** 针对 **`4680` 的专用 `q` 分支**（工程内 **`q.java` 无 `4680`**）。成功时服务端一般会 **照常下发场景/角色同步包**，由已有逻辑消化，例如（与 `docs/网络包协议分析.md` 一致）：

| 可能出现的包 ID | 典型作用 |
|-----------------|----------|
| **8192 (`0x2000`)** | 批量子包容器 |
| **8199 (`0x2007`)** | **`bt.c`**：地图 id、地图名、角色坐标等摘要 |
| **8200 (`0x2008`)** | **`bt.b`**：地图点集等 |
| **8197 (`0x2005`)** | **`bt.d`**：角色属性大段刷新（若服务端全量推） |
| **8198 (`0x2006`)** | 地图实体批次（`az[]`，见 §9） |
| 其他 | 视服务端实现可能还有传送相关 UI 包等 |

客户端表现上：**角色被切回城市场景、坐标与地图名更新、背包中回城券数量随后续刷新包变化**；**不会出现** 本条「回城券数量不足」的 8193 自定义 UTF（除非服务端仍先发一条成功提示，取决于服务端配置）。

## 8. `8199` 「刷新地图」 和回城逻辑hook

### 8.1 语义说明

**可以粗理解为「地图/位置摘要包」**：收到 **`0x2007`（8199）** 后，`q.b()` 会先执行 **`bt.c`**，把 **当前地图 id、地图名、角色坐标、4 个 byte 标志** 写入全局静态字段；再执行 **换图判断、更新 `a.e.I` 上的逻辑坐标、清队列、调用 `a.e.I.d()` / `a.e.e()` 等**。  
因此 **8199 是客户端切场景、同步站立点的主入口之一**，但 **地表贴图/阻挡等** 往往还要靠 **8200（点集）、8260（地图块）** 等后续包；若只注入 8199，可能出现 **坐标已变但地图资源未加载或错位**，需按实际效果决定是否再伪造或等待其它包。

### 8.1.1 实测崩溃原因：只 fake `8199` 会触发空地图对象

一次实测中，按 **「空间通道 8199」+「傲来广场 8199」** 的方式 fake 收包后，客户端成功进入 `q.b()` 的换图分支，但随后在：

- `q.b()` → `a.e.e()`
- `ao.e()` → `ar()`
- `ao.ar()` → `a.f.a(this.f, this.e.m)`

这条链路上触发了 **`NullPointerException`**。

根因是：`ao.ar()` 会先尝试通过 **`bt.a(bt.ar)`** 读取目标地图缓存；若缓存中没有，再退回 **`a.ae.c(String.valueOf(bt.ar))`** 从当前资源管理器加载地图。若两者都失败，则 **`this.f == null`**，而 `ai.a(aw, int)` 会直接访问地图对象字段，最终空指针崩溃。

结论：

- **`8199` 足以驱动客户端进入换图逻辑**
- 但 **`8199` 不保证目标地图资源已经到位**
- 若目标地图未缓存、也未被当前资源管理器持有，则切图阶段会直接崩溃

### 8.2 `fakePayload` 字段与二进制格式

与 **`bt.c(DataInputStream)`**（`bt.java` 约 1143～1151 行）严格一致，按 **`DataInputStream` 读序** 构造 **`byte[]`**（全大端）：

| # | 方法等价 | 目标字段 | 类型 |
|---|-----------|----------|------|
| 1 | `readShort` | `bt.ar` | `short`，地图 id |
| 2 | `readUTF` | `bt.ae` | Java UTF：`uint16` 正文长度 + Modified UTF-8 正文 |
| 3 | `readShort` | `bt.at` | `short`，X |
| 4 | `readShort` | `bt.au` | `short`，Y |
| 5 | `readByte` | `bt.av` | `byte` |
| 6 | `readByte` | `bt.aw` | `byte` |
| 7 | `readByte` | `bt.bw` | `byte` |
| 8 | `readByte` | `bt.bx` | `byte` |

**长度公式**：`payloadLen = 12 + N`，其中 **`N` 为地图名字符串的 UTF-8 编码字节数**（BMP 汉字一般为每字 3 字节；`writeUTF` 的长度域为 2 字节，不含自身）。

**推荐构造方式**（与游戏内写出一致，避免手搓 hex）：

```java
ByteArrayOutputStream bos = new ByteArrayOutputStream();
DataOutputStream out = new DataOutputStream(bos);
out.writeShort(mapId);           // bt.ar
out.writeUTF(mapName);           // bt.ae
out.writeShort(x);               // bt.at
out.writeShort(y);               // bt.au
out.writeByte(av);
out.writeByte(aw);
out.writeByte(bw);
out.writeByte(bx);
out.flush();
byte[] fakePayload = bos.toByteArray();
```

将 **`fakePayload`** 作为 **`w.b`**，**`id = 8199`**，通过 **`av` 的收包入队**（如反射调用私有 **`b(w)`**）交给 **`q.a(w)`** 处理。

**报文 A**（24 字节）— 适合对照「短地图名 + 负坐标」：

| 十六进制片段 | 含义 |
|--------------|------|
| `00 01` | `bt.ar = 1` |
| `00 0C` + `E7…93`（12 字节） | `bt.ae = "空间通道"` |

| `FF F6` | `bt.au = -10` |
| `00 00 00 00` | `bt.av`～`bt.bx` 均为 0 |

**报文 B**（21 字节）— 主城类示例：

回城目标若与服务器一致，可直接 **复用一次真实成功回城时抓到的整段 `w.b`（8199）**，无需自己算 UTF-8；若自建坐标，**至少保证 `mapId/mapName` 与客户端已缓存资源匹配**，否则易出现黑图、挡墙缺失等问题。

### 8.3.1 `回城分析日志1.out` 中观察到的更完整回城序列

对 `回城分析日志1.out` 中多组真实换图片段梳理后，可归纳出比“双 8199”更完整的常见模式：

1. 前置出现 **`8245 (0x2035)`**，其 `readShort` 首字段与目标地图 id 对应，如 `1 | -1`、`14 | -1`、`20 | -1`、`15 | -1`、`4 | -1`、`3 | -1`
2. 随后进入 **`8199(空间通道)`**
3. 再下发 **`8199(目标图)`**
4. 紧跟 **`8198`**，刷新该地图上的 NPC / 实体摘要
5. 最后跟 **`8200`**，刷新地图点集/关键坐标

已确认的真实样本包括：

- `傲来广场`：`8245(14|-1)` → `8199(空间通道)` → `8199(傲来广场 240,304)` → `8198(303 bytes)` → `8200(9 bytes)`
- `龙宫广场`：`8245(20|-1)` → `8199(空间通道)` → `8199(龙宫广场 336,256)` → `8198(511 bytes)` → `8200(17 bytes)`
- `长寿村`：`8245(4|-1)` → `8199(空间通道)` → `8199(长寿村 160,336)` → `8198(334 bytes)` → `8200(13 bytes)`
- `小木屋`：`8245(3|-1)` → `8199(空间通道)` → `8199(小木屋 128,288)` → `8198(37 bytes)` → `8200(5 bytes)`

注意：现有 `回城分析日志1.out` 中的 **`8245`** 十六进制正文被截断，当前只能确认其 **地图 id / 替换 id / gzip 资源块** 结构与存在性，**还不能直接原样抄成完整 fake 数据**。

### 8.3.2 `回城分析日志2.out` 抓包时的角色移动路径

生成 `回城分析日志2.out` 时的实际操作顺序为：

1. 启动游戏，角色在 **长寿村**  
2. 前往 **豆仙庭**  
3. 前往 **傲来广场**  
4. 前往 **龙宫**（文档与样本里多记为 **龙宫广场**，`mapId` 与傲来等并列出现）  
5. 再回到 **豆仙庭**  
6. 再回到 **长寿村**

对照解读日志时请注意：

- 文件中 **`8245` / 双 `8199` / `8198` / `8200`** 等成段出现，通常对应**某一次完整切图**（例如最后一次回到长寿村），而不是把上述六段路程各自对应六段连续、等长的包序列；需在日志时间轴上按 `[NET] 接收` 分段查找。
- 与本文及抓包样本中已出现的 **`8199` / `8245` 首字段** 对照（非全地图枚举）：**长寿村 ≈ `4`**、**傲来广场 ≈ `14`**、**龙宫广场 ≈ `20`**、**豆仙庭 ≈ `78`**（见 §8.3 报文 B）。
- `回城分析日志2.out` 中回到长寿村的一段里，**`8198` 曾出现在两条 `8199`（空间通道 → 目标图）之前**，与 §8.3.1 归纳的「先双 `8199` 再 `8198`」**不完全一致**；说明服务端下发顺序可能因版本或场景略有差异，fake 时仍以**真实抓包顺序**为准更稳妥。

### 8.4 与 `8193` 同时到达时的 Hook 顺序建议

若服务端先回 **8193（失败文案）** 再回其它包：应在 **`q.a(w)`** 或 **`av` 入队前** **丢弃/改写** 该失败 **8193**，再注入 **8199**，避免 **`a.b(String)`** 先弹窗；若只注入 **8199** 而不屏蔽 **8193**，用户仍会看到失败提示。

### 8.5 待优化项

- **补齐真实回城资源序列**：为每个目标地点重新抓取未截断的 **`8245`** 完整 payload，必要时连同 **`8260`** 一起归档，再将 hook 从“只 fake `8199`”升级为“按地点复用完整真实收包序列（`8245 -> 8199 -> 8199 -> 8198 -> 8200`）”。

## 9. 四类包：语义、解析链路与构造 fake 的依据

以下四类包是「切到一张**客户端尚未缓存**的地图」时最常一起出现的收包；含义与**怎么读 payload**完全由 `q.a(w)` 分派到的解析函数决定。只有对齐这些语义，才能判断 fake 时要带哪些字段、哪些包可以省略或必须复用真包。

| 包 ID（十六进制） | `q` 分支 | 核心解析 | 在客户端里的作用（概括） |
|------------------|----------|----------|-------------------------|
| **8245** `0x2035` | `case 8245` | `bt.aI` + 尾字节 | 按**地图 id**把一块**二进制资源**（多为 gzip）放进全局向量 `rk`，供 `bt.a(mapId)` **解压**后交给地图解码器 |
| **8199** `0x2007` | `case 8199` | `bt.c` + 私有 `b()` | 写入当前图 id/名/坐标等到 `bt.ar`…`bt.bx`，并驱动 **换图主流程**（`a.e.e()` / `f()`、`ar()` 等） |
| **8198** `0x2006` | `case 8198` | `az.a` × N | 本图 **NPC/地图实体** 列表；经 `bt.u` 缓冲，在 `ao.P()` 里变成 `bt.t` 并触发 **立绘/资源预载** |
| **8200** `0x2008` | `case 8200` | `bt.b` | 本图 **兴趣点坐标对**（`bt.ab[]` / `bt.ac[]`），用于小地图点、部分交互与绘制 |

### 9.1 `8245`：地图资源分块 — `bt.aI`

**`q` 入口**（`jadx/sources/defpackage/q.java`）在调用 `bt.aI(this.b)` 之后，再读**一个字节**：若为 `1`，则调用 `this.a.e()`（壳层 `com.yinhan.kjava.main.a`，具体行为需看主程序；与 defpackage 内逻辑解耦）。

**`bt.aI` 读包顺序**（`bt.java`）要点：

1. **`readShort` → `s2`**：资源块绑定的 **地图 id**（与 `br.a`、`bt.a(short)` 查找键一致；反编译里带调试输出 `s2 | s3`）。
2. **`readShort` → `s3`**：**替换/删除**用 id。若 **`s3 < 0`**（日志里常见 **`-1`**），只做增量添加，不从 `rk` 删旧块；若 **`s3 >= 0`**，会在后面从向量 `rk` 里删掉所有 **`br.a == s3`** 的项。
3. **正文是否读取**与当前缓存有关：
   - 若 **`rk` 为空**：必定 **`readInt` → 长度 `L`，再 `read` 满 `L` 字节**，新建 **`br(s2, bArr)`** 加入 `rk`，然后 **return**（不再处理 `s3` 删除）。
   - 若 **`rk` 非空**且 **尚不存在 `br.a == s2`**：同样 **`readInt` + `read` `L` 字节**，加入新块。
   - 若 **已存在同 `s2` 的块**：**不再读取** `L` 与后续字节（协议上应理解为「同 id 重复包不带体」或仅发头；否则流会错位，实际以服务端为准）。
4. 若 **`s3 >= 0`**：自尾向前扫描，删除 `br.a == s3` 的块。

**解压与使用**：`bt.a(short s2)` 在 `rk` 里找到 `br.a == s2` 的 `byte[]`，经 **`ci.a`（gzip 解压）** 得到地图用的裸数据。换图时 **`ao.ar()`** 会 **`byte[] bArrA = bt.a(bt.ar)`**；若非空则用 **`a.ae.a(bArrA)`** 解析成当前地图对象 **`this.f`**，否则退回 **`a.ae.c(String.valueOf(bt.ar))`** 从包内资源加载。仅 fake `8199` 而不先有 **`8245` 写入 `rk`** 时，`bt.a(bt.ar)` 常为 **`null`**，若本地也无该 id 资源，即 **`this.f == null`** → 后续渲染/逻辑空指针。

**构造 fake 的结论**：`8245` 的 payload **不能只写 id**，必须带 **`int` 长度 + 与真服一致的 gzip 体**（或保证客户端已通过别途径让 `bt.a(mapId)` 能返回合法数据）。首包前 **`rk` 状态**决定是否会读体，重放抓包时注意顺序与是否清过缓存。

### 9.2 `8199`：地图/位置摘要 — `bt.c` 与 `q.b()`

**`q` 入口**：`case 8199` 仅调用私有方法 **`b()`**。

**`b()` 内第一步**：**`bt.c(this.b)`**，按 **`DataInputStream`** 顺序写入静态字段：

- `bt.ar` `short`：地图 id（与 `8245` 的 `s2`、`bt.a` 查找键一致）
- `bt.ae` `String`：地图名
- `bt.at` / `bt.au` `short`：角色像素坐标
- `bt.av`～`bt.bx` 各 `byte`：朝向/标志等（换图分支里会用到如 `bt.aw`）

随后 **`b()`** 根据 **`a.e.Y` 与 `bt.ar` 是否变化**、队伍状态等，调用 **`a.e.e()`**（完整换图）或 **`a.e.f()`**（同图刷新）、**`a.e.I.d()`** 等；其中 **`a.e.e()`** 会走到 **`ao.ar()`**，从而触发上一节的 **`bt.a(bt.ar)`** 加载地图。

**构造 fake 的结论**：`8199` 的 **二进制布局**就是 §8.2 的表；但要避免崩溃，**`bt.ar` 必须与已存在的地图数据一致**（`rk` 里块或内置资源）。双 `8199`（空间通道再目标图）是服务端习惯，客户端对**每条**包各执行一次完整 `b()`，没有「必须两条」的硬编码状态机。

### 9.3 `8198`：地图实体批次 — `az.a` 与 `bt.u` / `bt.t`

**`q` 入口**（逻辑概要）：

1. **`readByte` → `n`**（有符号字节；一般 `n > 0`）。
2. 若 **`n > 0`**：分配 **`az[n]`**，每个元素调用 **`az.a(DataInputStream)`** 读满一个实体，然后 **`bt.u.addElement(整表数组)`**（一次包对应**一个** `Vector` 元素，类型为 `az[]`）。
3. **`ao.Q()`**：把静态标志 **`cL = true`**，表示「有待处理的实体表」。
4. 若 **`a.e != null`**：**`a.e.P()`**。

**`az.a` 读序**（`az.java`，字段含义为 NPC/怪物名、格子坐标、多层 bit 标志、可选 UTF 与字符串表等）：依次为 `readShort`（实体 id）、`readUTF`（显示名）、`readShort`（丢弃）、`readShort`→`c`、`readShort`→`d`（坐标经 `+8` 偏移）、再 `readShort`（丢弃），随后 **多字节 bit 打包**、`readByte`/`readShort` 若干，以及可选 `readUTF`、成对 `readUTF` 数组、`readInt` 等。**单条实体长度不固定**，取决于标志位；**fake 时强烈建议直接复用抓包 hex**，不要手搓。

**`ao.P()`**（在适当时机被调用）：若 **`bt.u` 非空**，取出**最后一个**元素赋给 **`bt.t`**（当前地图实体表），并 **`bt.u.removeAllElements()`**；再为各实体 **`a.ah.a(...)` 预载立绘/外形**、赋 **`bt.t[i].w`** 等。若 **`8198` 缺失或 payload 错误**，常见表现是 **NPC 不显示或资源错乱**，一般不如缺 `8245` 那样直接导致 **地图对象 null**；是否与空指针相关取决于后续是否访问未初始化的实体字段。

**`n == 0` 时**：不向 `bt.u` 添加新表，但仍执行 **`ao.Q()`** 与 **`P()`**（仅刷新标志与一次合并逻辑）。

### 9.4 `8200`：地图点集 — `bt.b`

**`bt.b(DataInputStream)`**：

1. **`readByte` → `n`**（点数）。
2. 分配 **`bt.ab = new short[n]`**、**`bt.ac = new short[n]`**。
3. 循环 **`n` 次**：**`ab[i] = readShort()`**，**`ac[i] = readShort()`**。

**使用处**：如 **`ao`** 绘制小地图上的点、**`ac`** 中与坐标相关的判定等。与 **阻挡/主图层**无直接替代关系；缺了可能 **小地图点缺失**，一般不单独导致 **`this.f` 为空**。

**构造 fake 的结论**：格式极简（`1 + 4n` 字节量级）；可与真包一致复写，或 **`n=0`** 表示清空点集（需确认客户端对 `null`/空数组的后续判断是否与你的版本一致）。

### 9.5 综合：为何 fake 要「四类一起考虑」

| 需求 | 依赖 |
|------|------|
| 换图后不 NPE、有地形可走 | **`8199` 的 `bt.ar`** 必须在 **`ao.ar()`** 里能 **`bt.a(bt.ar)` 成功或 `a.ae.c` 成功** → 通常依赖 **`8245`** 提前把该 id 的 gzip 块放进 **`rk`**，或客户端包内自带该图 |
| 地图上 NPC/交互正常 | **`8198`** 提供 **`bt.t`** 及 **`P()`** 里触发的资源加载 |
| 小地图/兴趣点一致 | **`8200`** 提供 **`bt.ab`/`bt.ac`** |

**下发顺序**：源码**没有**把四类包绑成固定状态机；`8198` 与双 `8199` 的先后顺序以**真实抓包**为准（见 §8.3.2）。**最稳妥的 fake** 仍是：**按一次真实切图，原样重放 `8245`（及尾字节语义）+ 若干 `8199` + `8198` + `8200` 的字节序与内容**。
